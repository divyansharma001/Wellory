# Health Tracker Server

An AI-powered backend for a health tracking application that combines:

- structured log storage in PostgreSQL
- asynchronous processing with Redis and BullMQ
- semantic memory in Qdrant
- AI-powered embeddings, fact extraction, daily summaries, and health coaching with Gemini

The system keeps the same core architecture as the earlier productivity-coach backend, but the domain is now health tracking: meals, calories, exercise, symptoms, sleep, food sensitivities, and personalized coaching.

## What This Server Does

The backend supports three main jobs:

1. User authentication
2. Health log ingestion and background enrichment
3. AI chat grounded in the user's stored health history

When a user records a health log, the server:

- stores the raw log in PostgreSQL
- queues the log for processing
- embeds the log into vector memory
- extracts health facts from the log
- updates a daily health summary

When a user asks a question in chat, the server:

- embeds the question
- retrieves relevant facts, past logs, and today's summary
- ranks and filters that context
- generates a health-coach response
- runs a verification pass before returning the answer

## Architecture Overview

The application is built around a small set of cooperating components:

- `Express`: HTTP API and request handling
- `Better Auth`: user authentication and session management
- `PostgreSQL + Drizzle ORM`: source of truth for auth data and raw health logs
- `Redis + BullMQ`: background job queue for log enrichment
- `Qdrant`: vector database for semantic memory retrieval
- `Gemini`: embeddings, fact extraction, daily summaries, and grounded chat generation

### High-Level Flow

```text
Client
  -> Express API
    -> Better Auth session check
    -> PostgreSQL stores raw health log
    -> BullMQ queues processing job
      -> Worker consumes job
        -> Gemini creates embeddings and extracts facts
        -> Qdrant stores logs, facts, and daily summary
        -> PostgreSQL log status becomes processed

Client chat question
  -> Express API
    -> Better Auth session check
    -> Gemini embedding for the question
    -> Qdrant retrieves relevant facts, logs, and daily summary
    -> Context ranking and conflict handling
    -> Gemini generates answer
    -> Gemini verifies answer against retrieved context
    -> Response returned to client
```

## Cognitive Memory Model

The chat system uses three memory layers:

- `Long-term memory`: extracted user facts stored in the `user_facts` Qdrant collection
- `Working memory`: the current day's health summary stored in `daily_summaries`
- `Episodic memory`: individual historical logs stored in `health_logs`

This structure lets the server answer health questions using:

- durable user patterns
- what happened today
- specific relevant historical events

## Core Components

### 1. API Layer

File: [src/index.ts](C:\health-tracker\server\src\index.ts)

Responsibilities:

### 2. Authentication

File: [src/lib/auth.ts](C:\health-tracker\server\src\lib\auth.ts)

Authentication is handled by Better Auth using a Drizzle adapter. Auth data is stored in PostgreSQL using the same schema used by the application.

Tables involved:

- `user`
- `session`
- `account`
- `verification`

### 3. Database Layer

Files:

- [src/db/index.ts](C:\health-tracker\server\src\db\index.ts)
- [src/db/schema.ts](C:\health-tracker\server\src\db\schema.ts)

PostgreSQL is the system of record for:

- users and sessions
- raw health logs
- log processing status

The `log_entry` table stores:

- `id`
- `user_id`
- `content`
- `type`
- `status`
- `created_at`
- `updated_at`

The default log type is `health_log`.

### 4. Queue and Worker

Files:

- [src/lib/queue.ts](C:\health-tracker\server\src\lib\queue.ts)
- [src/lib/redis.ts](C:\health-tracker\server\src\lib\redis.ts)
- [src/workers/log.worker.ts](C:\health-tracker\server\src\workers\log.worker.ts)

When a new health log is submitted, the API stores it immediately and enqueues a `process-log` job.

The worker then:

1. creates an embedding for the raw log
2. stores the log in Qdrant as episodic memory
3. extracts health facts from the log
4. embeds and stores those facts in Qdrant
5. updates the current day's summary
6. marks the database row as `processed`

If anything fails, the worker marks the log as `failed`.

### 5. Vector Memory

File: [src/lib/qdrant.ts](C:\health-tracker\server\src\lib\qdrant.ts)

Qdrant stores three collections:

- `health_logs`
- `user_facts`
- `daily_summaries`

Each vector search is filtered by `userId`, so a user's memory is isolated from other users.

### 6. AI Layer

File: [src/lib/gemini.ts](C:\health-tracker\server\src\lib\gemini.ts)

Gemini is used for four jobs:

- `generateEmbedding(text)`: convert a log or chat query into a 3072-dimension vector
- `extractFacts(text)`: pull out health-specific facts from a user log
- `updateDailySummary(currentSummary, newLogEntry)`: maintain a daily health narrative
- `generateVerifiedResponse(prompt, contextData)`: generate a chat answer, then verify it against source context

This design helps the model stay grounded in actual retrieved user history instead of answering from generic assumptions alone.

## Health Log Processing Flow

### Step 1. User submits a log

`POST /api/logs`

Example:

```json
{
  "content": "Had a 600 calorie salad for lunch and felt bloated after."
}
```

### Step 2. API stores raw log

The API:

- validates that the user is authenticated
- inserts the log into PostgreSQL with status `pending`
- enqueues a background job

### Step 3. Worker enriches the log

The worker:

- stores the raw log in `health_logs`
- extracts facts like:
  - "User felt bloated after this meal"
  - "User logged a 600 calorie lunch"
- updates the daily summary for that user

### Step 4. Log status is updated

The row in `log_entry` is updated from `pending` to `processed`.

## Chat Response Flow

`POST /api/chat`

Example:

```json
{
  "message": "What did I eat today?"
}
```

The chat route:

1. checks the authenticated session
2. embeds the user message
3. searches Qdrant for:
   - relevant facts
   - relevant historical logs
   - today's summary
4. scores facts using similarity and recency
5. detects conflicts between stored facts
6. builds the health coach system prompt
7. generates a draft answer
8. verifies the draft against the retrieved context
9. returns the final answer

## Context Engineering

The chat system does more than simple vector retrieval. It performs a small ranking pipeline before building the final prompt.

### Fact Scoring

Facts are ranked by:

- vector similarity
- recency decay

The combined score is:

```text
combinedScore = similarity * recencyWeight
```

This lets the system prefer recent, relevant facts over stale ones.

### Fact Deduplication

Facts are normalized into keys using token filtering. This reduces duplicates such as semantically similar statements about the same recurring health pattern.

### Conflict Detection

The system compares fact polarity to catch contradictions such as:

- "User is allergic to peanuts"
- "User tolerates peanuts"

When conflicting facts appear, the conflict is surfaced into the prompt so the model can answer more carefully.

## Current Design Strengths

- clear separation between transactional storage and semantic memory
- fast user response because log enrichment runs asynchronously
- grounded chat using retrieved facts and history
- conflict-aware prompting for safer personalized responses
- domain pivot achieved without changing the underlying architecture

## Current Limitations

- much of the orchestration still lives in `src/index.ts`
- request validation is minimal even though `zod` is installed
- there are no automated tests yet
- health guidance is coaching-oriented, not medical diagnosis
- Qdrant host and port are currently hardcoded in `src/lib/qdrant.ts`

## Suggested Next Improvements

1. split `src/index.ts` into route and service modules
2. add request validation with Zod for all POST routes
3. add tests for worker behavior and fact conflict detection
4. move Qdrant configuration into environment variables
5. add rate limiting and structured error handling in the API layer
6. add observability for queue failures and AI latency

## Notes

- This project is a health tracking and coaching backend, not a medical device.
- AI answers should be treated as supportive guidance, especially for nutrition and habit tracking.
- For medical symptoms, allergies, or urgent health concerns, the application should not replace professional care.
